   Risk: Tight coupling, hard to test, performance issues as data grows

   Location: src/analysis/orchestrator.ts (320 lines)
   Problem: Handles 7 different responsibilities

   Solution: Extract file handling and KMS coordination to separate modules
   Effort: 4-5 hours (HIGH)
   
Issue #3: Type Duplication
   Location: src/types.ts and app/types.ts
   Problem: InferredRelationship defined in both files
   Risk: Type divergence, maintenance nightmare
   Solution: Delete app/types.ts, import from src/types.ts
   Effort: 1 hour (HIGH)

================================================================================
DECISION RATIONALES
================================================================================

✓ Manifest-Based Caching
  Why: Offline operation, no external DB, fast lookups
  Trade-off: Must rebuild if corrupted (mitigated by auto-recovery)

✓ Per-Model Analysis Cache
  Why: Supports user model flexibility (Haiku then Opus)
  Trade-off: Manifest size grows with models

✓ File System State
  Why: Simple, no external dependencies
  Trade-off: File I/O for every API call

✓ Three Separate Commands
  Why: Flexibility for power users (convert separately)
  Trade-off: CLI code duplication (fixable)

✓ Multi-Agent Pattern
  Why: Separation of concerns, specialization
  Trade-off: Requires orchestration complexity
