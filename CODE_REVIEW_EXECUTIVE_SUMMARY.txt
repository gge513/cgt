   Files: app/api/kms/{decisions,summary,relationships,actions}.ts
   Impact: Type safety collapse - defeats TypeScript purpose
   Time: 2 hours

   Example:
   ❌ const decisions: any[] = [];
   ✅ const decisions: KMSDecision[] = [];

2. FIX VALIDATION IN RELATIONSHIP PARSER
   File: src/kms/relationshipInferencerDSPy.ts (lines 157-160)
   Impact: Invalid union types could crash UI
   Time: 1 hour

   Example:
   ❌ return { fromType: rel.fromType as any }
   ✅ if (!isValidItemType(rel.fromType)) return null;

3. ADD RESPONSE TYPE CONTRACTS
   Files: All API routes
   Impact: API contract fragility
   Time: 1.5 hours

   Example:
   ❌ return NextResponse.json({ ... })
   ✅ const response: KMSSummaryResponse = { ... };

   File: app/api/kms/summary/route.ts (line 68)
   Impact: Returns NaN to client
   Time: 15 minutes

   Example:

KIERAN'S RECOMMENDATION
================================================================================

DO NOT DEPLOY TO PRODUCTION without fixing critical issues.

The widespread use of `any` types creates false type safety while actually
disabling TypeScript's protections. The relationship inference validation
is particularly critical—invalid types from Claude could crash the UI.

RECOMMENDED APPROACH:
  1. Fix CRITICAL issues this sprint (1-2 days)
  2. Address HIGH issues in parallel
  3. Add comprehensive tests
  4. Merge after code review

GOOD NEWS:
  - Architecture is fundamentally sound
  - These are implementation issues, not design problems
  - With fixes, this will be a robust, maintainable system
