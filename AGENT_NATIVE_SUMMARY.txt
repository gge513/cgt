CRITICAL GAPS EXPLAINED
================================================================================

GAP 1: FILE UPLOAD
=================
Problem: Agent cannot programmatically add transcripts
Current: Agent must use bash: cp /tmp/meeting.txt input/
Needed: POST /api/kms/transcripts { filename, content }
Impact: Blocks autonomous operation (agent can't source transcripts)
Fix Time: 2-3 hours
After Fix: Agent can upload transcripts and trigger analysis without bash

GAP 2: ANALYSIS TRIGGER
======================
Problem: Analysis functions not API-exposed, only CLI accessible
Current: Agent must exec("npm run analyze") and parse exit codes
Needed: POST /api/analysis/analyze { forceReprocess?: boolean }
Impact: Indirect error handling, no progress visibility
Fix Time: 3-4 hours
After Fix: Structured API response with stats and timing

GAP 3: STATE INSPECTION
======================
Problem: Agent cannot check readiness before operating
Current: Agent must bash: cat .processed_manifest.json | jq .
Needed: GET /api/analysis/state returns file counts, API config, cache stats
Impact: Blind operation (doesn't know if system is ready)
Fix Time: 1-2 hours
After Fix: Agent knows readiness, file counts, model, cache state

================================================================================
ARCHITECTURE STRENGTHS
================================================================================

1. Non-Interactive CLI
   - No prompts, fully scriptable
   - Clear semantics: convert vs analyze vs analyze-existing
   - Powerful query language for KMS

2. Manifest Pattern
   - Single source of truth: .processed_manifest.json
   - Per-model caching (Haiku vs Opus separate)
   - Atomic writes prevent corruption

3. Specialist Agents
   - Separation of concerns (synthesizer, strategist, impact analyst)
   - Focused prompts = better outputs
   - Composable analysis results

4. Error Handling
   - Graceful degradation per file
   - Detailed logging at boundaries
   - Clear failure modes

================================================================================
ARCHITECTURE WEAKNESSES
================================================================================

1. Headless Workflow (CLI-only)
   - Analysis only accessible via subprocess
   - No structured API for orchestration
   - Progress not visible to agent

2. File Operations Manual
   - No upload API (agents must bash cp)
   - No file listing API
   - No file deletion API
   - No file move/organize API

3. State Inspection Manual
   - Manifest must be JSON-parsed manually
   - No API to check file counts
   - No API to check API key configuration
   - No API to view cache statistics

4. UI-API Mismatch
   - Drill-down detail not in API
   - Keyword search only in CLI
   - Single-item endpoints missing
